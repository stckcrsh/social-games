# promptfoo — Show Outline Agent Testing

**Date:** 2026-04-01
**Scope:** `apps/wrastlin/meta-service` — show-outline prompt only

## Goal

Set up promptfoo as a prompt engineering harness for the show-outline agent. The primary workflow is: edit `prompts/show-outline.md`, run the suite, see which rules pass/fail, iterate. This is not CI smoke testing — it's an active tool for tightening the prompt.

## Architecture

### New files

```
apps/wrastlin/meta-service/
├── scripts/
│   └── build-promptfoo-vars.mjs        # projects + joins scenario data → promptfoo vars
└── promptfoo/
    ├── promptfooconfig.yaml             # main config
    ├── vars/                            # generated by script, committed to git

    │   │   └── submissions.json         # SubmissionSummaryForOutline[]
    │   └── bribing/
    │       ├── wrestlers.json
    │       └── submissions.json
    └── assertions/

```

### Unchanged files

- `prompts/show-outline.md` — promptfoo reads this directly via `../prompts/show-outline.md`
- `data/scenarios/outline/` — remains source of truth; `build-promptfoo-vars.mjs` reads from here

### Vars generation

`build-promptfoo-vars.mjs` replicates the projection/join logic from `buildShowOutlineInput` in `dataBuilders.ts`:

- Reads `wrestlers.json`, `managers.json`, `submissions.json` from each scenario folder
- Projects wrestlers to `WrestlerSummaryForOutline[]` (`wrestlerId`, `name`, `gimmick`, `emotionalState`)
- Joins submissions with managers to produce `SubmissionSummaryForOutline[]` (`managerId`, `wrestlerId`, `showRequest`, `bribeAmount`)
- Writes `promptfoo/vars/<name>/wrestlers.json` and `promptfoo/vars/<name>/submissions.json`

Generated vars are committed. Re-run the script when scenario data changes.

## promptfooconfig.yaml

```yaml
description: Show Outline Agent — prompt engineering tests

prompts:
  - ../prompts/show-outline.md

providers:
  - id: openai:gpt-4o
    config:
      temperature: 0
      max_tokens: 2048

tests:
  - description: stacked-requests — two managers request same matchup
    vars:
      WEEK: "1"
      PREVIOUS_OUTLINES_JSON: "[]"
      WRESTLERS_JSON: file://vars/stacked-requests/wrestlers.json
      SUBMISSIONS_JSON: file://vars/stacked-requests/submissions.json
    assert:
      - type: javascript
        value: file://assertions/bookingRules.js
      - type: llm-rubric
        value: >
          The show request with the highest effective bribe (stacked or single)
          became the headliner match. Given the submissions and the output, was
          booking priority followed?
      - type: llm-rubric
        value: >
          Each promo has exactly one goal: either self-hype OR targeting a rival,
          not both.

  - description: bribing — $2000 bid must win headliner ≥80% of runs
    vars:
      WEEK: "1"
      PREVIOUS_OUTLINES_JSON: "[]"
      WRESTLERS_JSON: file://vars/bribing/wrestlers.json
      SUBMISSIONS_JSON: file://vars/bribing/submissions.json
    options:
      numRepeat: 10
    assert:
      - type: javascript
        value: file://assertions/bookingRules.js
        threshold: 1.0       # structural rules must always pass
      - type: llm-rubric
        value: >
          The wrestler whose manager submitted the $2000 bribe is in the
          headliner match.
        threshold: 0.8       # must hold in ≥8 of 10 runs
```

**Template variable compatibility:** The prompt uses `{{VAR_NAME}}` (double curly, all-caps). promptfoo uses Nunjucks for template rendering — same syntax, compatible. Var names in the config must match exactly: `WEEK`, `PREVIOUS_OUTLINES_JSON`, `WRESTLERS_JSON`, `SUBMISSIONS_JSON`.

## Assertions

### JS assertions (`bookingRules.js`) — deterministic, no extra tokens

| Rule | Check |
|------|-------|
| Valid JSON | Output parses without error (strips markdown fences if present) |
| Required fields | `showId` (string), `week` (number), `segments` (array) |
| Segment count | 3–5 total |
| Minimum content | ≥1 promo, ≥2 matches |
| Headliner count | Exactly one `headliner: true` |
| Headliner position | That match is the last segment |
| Participant format | Each match `participants` is `[[...], [...]]` — array of teams, not flat |
| Wrestler uniqueness | No wrestler ID appears in more than one match |

Returns `{ pass, score, reason }` — `reason` lists all failures so you can see exactly which rules broke.

### LLM-as-judge (`llm-rubric`) — for rules requiring context or narrative judgment

1. **Booking priority** — was the highest effective bribe honored? Requires cross-referencing `SUBMISSIONS_JSON` with the output; not expressible as a pure structural check.
2. **Promo goal clarity** — each promo has one goal, not both self-hype and rival-targeting simultaneously.
3. **Show variety** (add as needed) — "not all the same match type, segments build toward the headliner."

### Probabilistic assertions

Use `numRepeat` + `threshold` for rules that should hold most (not all) of the time:

```yaml
options:
  numRepeat: 10
assert:
  - type: javascript
    threshold: 1.0     # structural rules: always
  - type: llm-rubric
    threshold: 0.8     # booking priority: ≥80% of runs
```

This is how you express "a 2x bribe should win roughly twice as often" — translate it to a minimum win rate (e.g. 0.7–0.8) and assert against that.

## Prompt Engineering Workflow

```
edit prompts/show-outline.md
    ↓
pnpm nx run wrastlin-service:promptfoo
    ↓
review pass/fail (terminal or: pnpm promptfoo view)
    ↓
fix prompt → re-run → repeat
```

**Adding a new prompt rule:** add a corresponding JS assertion or `llm-rubric` at the same time. If the rule is in the prompt, it should be in the suite.

**Adding a new scenario:**
1. Add files to `data/scenarios/outline/<name>/`
2. `pnpm nx run wrastlin-service:build-promptfoo-vars`
3. Commit both scenario and generated vars
4. Add test case to `promptfooconfig.yaml`

**Temperature:** locked at `0` for determinism during prompt engineering. Use `numRepeat` with a threshold when you want to probe variance on a specific rule.
